# TDD（テスト駆動開発）ルール

## TDDサイクル

### 1. Red（レッド）
失敗するテストを書く → `cargo test` で失敗を確認

Rustでは未実装の関数を呼ぶテストはコンパイルエラーになる。**コンパイルエラーもRed（失敗の確認）として扱ってよい**。アサーション失敗まで確認できる場合はそちらが望ましい。

### 2. Green（グリーン）
テストが通る最小限のコードを書く → `cargo test` で成功を確認

「最小限」はテストが要求しない機能を先回りで実装しないという意味。enumのバリアントやフィールドも、テストが必要とした時点で追加する（YAGNI）。

### 3. Refactor（リファクタ）
テストの成功を維持したままコードを改善する → `cargo test` で成功を確認

このフェーズで行うこと：
- 重複の排除、関数分割
- マジックナンバーの定数化
- テストコード自体の整理（ヘルパー抽出、パラメータ化）も対象
- `cargo fmt` と `cargo clippy --all-targets -- -D warnings` の解消

改善点がないサイクルではスキップしてよい。

## サイクルの粒度

**1サイクル = 1つの振る舞い（1テストケース）** を基本とする。複数のテストを一括で先に書かない — 一括で書くとGreenの「最小限」が判断できず、失敗原因の特定も難しくなる。

## 境界値テストの扱い

実装完了後に追加する境界値・網羅テスト（`testing` スキル参照）は最初から成功するためRedにできない。これらは**最後のRefactorフェーズで回帰テストとして追加**する（Redの成立を求めない）。

## 適用範囲

- 新規実装・機能追加はTDDサイクルで進める
- **既存コードへのテスト追加**はTDDサイクル外の作業。現在の振る舞いを固定する特性テストとして追加する
- `cargo init` が生成する雛形コード（`add` 関数と `it_works` テスト）は最初のRedの前に削除してよい
