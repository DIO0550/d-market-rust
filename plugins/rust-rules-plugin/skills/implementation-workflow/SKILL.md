# Rust実装ワークフロー

Rust開発における実装フローのエントリーポイント。状況に応じて適切なスキルを参照する。

## 関連スキル参照ガイド

| 状況 | 参照スキル |
|:-|:-|
| コーディング中・設計判断時 | `coding-standards` |
| テスト作成時 | `tdd`, `testing` |
| コミット時 | 本スキルの「コミット規約」セクション |

## 実装フロー

### 1. 作業開始

```bash
# mainで作業しない！必ず新ブランチを作成
# ベースは最新のmain（リモートがある場合は git pull origin main で最新化してから切る）
git checkout -b feature/機能名
```

git リポジトリが未初期化の場合は `git init -b main` で初期化し、雛形（cargo init の生成物）を初期コミットしてからブランチを作成する。

### 2. 実装計画を提示

コード変更前に以下を提示し承認を得る：
- タスクの理解と分析
- 実装すべき機能・モジュールの概要
- ファイル構成と変更対象
- 実装手順とステップ

承認者がいない自律実行環境（バックグラウンドタスク等）では、計画を出力に明示した上で続行してよい。ただしタスクの指示と矛盾する設計判断が必要になった場合は中断して確認する。

### 3. TDDで実装

→ 詳細は `tdd` スキルを参照

- Red → Green → Refactor のサイクル
- 先にテストを書いてから実装

### 4. コーディング規約に従う

→ 詳細は `coding-standards` スキルを参照

- 所有権と借用の適切な使用
- エラーハンドリング
- 命名規則

### 5. 品質チェック（必須）

```bash
cargo fmt                              # フォーマット
cargo clippy --all-targets -- -D warnings  # Lintチェック（テストコードも対象、警告をエラー扱い）
cargo test                             # テスト実行
```

**全て通るまでコミット禁止**

### 6. コミット

→ 下記「コミット規約」に従う

## コミット規約

Conventional Commits に従う。1コミット = 1つの論理的変更（雛形生成・リファクタ・機能追加を混ぜない）。

```
<type>: <概要（50文字以内）>

<必要なら本文（変更の理由・背景）>
```

| type | 用途 |
|:-|:-|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `refactor` | リファクタリング |
| `test` | テストのみの変更 |
| `docs` | ドキュメント更新 |
| `chore` | ビルド・設定・雛形など |

概要・本文の言語はプロジェクトの既存コミット履歴に合わせる（履歴がなければ日本語でよい）。

## 禁止事項

- `unsafe` の不必要な使用
- clippy警告の無視（`#[allow(...)]` 禁止）
- mainブランチで直接作業禁止
- `unwrap()` / `expect()` の本番コードでの使用

## ブランチ命名規則

| プレフィックス | 用途 |
|:-|:-|
| `feature/` | 新機能追加 |
| `fix/` | バグ修正 |
| `refactor/` | リファクタリング |
| `docs/` | ドキュメント更新 |

## チェックリスト

以下のチェックリストは作業中に自分で照合し、最終報告時に結果を要約してユーザーに提示する（ファイルとしては保存しない）。未達項目がある場合は理由を添えて報告する。

```
## 実装前
- [ ] 新ブランチを作成した
- [ ] 実装計画を提示し承認を得た

## 実装中
- [ ] TDDサイクルを守っている（tddスキル参照）
- [ ] コーディング規約に従っている（coding-standardsスキル参照）

## 実装後
- [ ] `cargo fmt` 実行済み
- [ ] `cargo clippy` 警告なし
- [ ] `cargo test` 通過
- [ ] コミットルールに従っている（commitスキル参照）
```
